# Алгоритм отрисовки архитектурной диаграммы (Arch Map)

Документ описывает **финальный** пайплайн от узлов/рёбер из данных до пикселей на canvas и экспорта. Реализация: `GraphCanvas.tsx`, `forceGraphLayout.ts`, `astarGridRouter.ts`, вспомогательные модули.

---

## 1. Входные данные и контекст

Компонент `GraphCanvas` получает:

| Параметр | Назначение |
|----------|------------|
| `nodes`, `edges` | Узлы C4 и рёбра графа (после фильтров на уровне вкладки). |
| `selectedNodeId` | Выбранный узел — влияет на **схлопывание в группы** и на **раскладку** (центральный слой и т.д.). |
| `focusNodeId` | «Корень» диаграммы — смещает подпись ребра к противоположному от фокуса концу. |
| `tagCountByNodeId` | Число локальных тегов на узле (полоска и бейдж). |

Тема (`light` / `dark`) берётся из `ThemeProvider`; цвета canvas **не из CSS**, а из палитры `getDiagramPalette()` (`diagramTheme.ts`).

---

## 2. Предобработка: схлопывание плотных групп (`collapseDenseSimilarNodes`)

**Условие активации:** задан `selectedNodeId`. Если выбора нет — граф не меняется, кластеров нет.

**Идея:** узлы одного типа (Container, SoftwareSystem, Component, ContainerInstance, DeploymentNode), которые связаны с выбранным узлом и имеют **одинаковую отображаемую подпись связи** (`graphEdgeLabel`), при числе ≥ `CLUSTER_MIN_SIZE` (5) объединяются в один **синтетический узел-кластер**.

- Строится `ClusterMeta`: список участников, тип, подпись ребра.
- В графе участники заменяются одним узлом с `properties.__cluster === true` и размерами из `clusterDimensions`.
- Рёбра переназначаются на id кластера, петли удаляются, дедупликация по паре и типу.

**Результат:** `visibleNodes`, `visibleEdges`, `Map` кластеров, `selectedVisibleId` (id кластера, если выбранный узел вошёл в группу).

## 3. Раскладка узлов (`createForceSimulation` в `forceGraphLayout.ts`)

Размер карточки: обычно `NODE_W × NODE_H` (кратно `GRID_CELL`), для кластера — из `__clusterWidth` / `__clusterHeight` с минимумами.

### 3.1. Специальный случай: Deployment / Environment

Если выбран узел с лейблом `DeploymentEnvironment` или `Environment`, вызывается **`createDeploymentEnvironmentTreeLayout`**: BFS от корня по рёбрам, уровни по глубине, горизонтальная раскладка строк. Симуляция d3-force с одной силой **collide**, `alpha(0)` — положения по сути фиксированы после инициализации.

### 3.2. Основной режим: слои (tiers) по типу C4

1. **`tierFromLabels`** сопоставляет узлу индекс слоя по порядку `LAYER_ORDER` (SoftwareSystem → Container → …).

2. Узлы группируются по слоям. Для каждого слоя вычисляются вертикальные полосы (`tierTops`, `tierHeights`) с учётом высоты содержимого слоя.

3. Внутри слоя: **`placeTierWithoutOverlap`**, затем **`relaxTierWithForce`** — локальная мини-симуляция d3 на строках слоя: `charge`, `collide` (радиус зависит от узла; для кластера — увеличенный зазор), `forceX` к центру; Y жёстко «пришивается» к строкам; несколько итераций `tick`.

4. Если есть выбранный узел и его слой известен, **поднимается и центрируется по X** выбранный узел относительно соседей в том же слое (`fx`/`fy`).

5. **`relaxTierHorizontalPreserveY`** — ещё один проход разведения по X с закреплением выбранного id при необходимости.

6. **`resolveTierHorizontalAABBCollisions`** и **`resolveCrossTierOverlap`** — детерминированное разведение пересекающихся прямоугольников (несколько итераций).

7. Координаты **переводятся из центра карточки в левый верхний угол** (`x -= width/2`, `y -= height/2`), т.к. отрисовка и A* работают в координатах **левого верхнего угла**.

8. Возвращается `forceSimulation` с **`alpha(0)` и без сил** (в этом пути) — т.е. основная «симуляция» в `GraphCanvas` не двигает узлы; тики служат для инвалидации кэша и перерисовки до остановки alpha.

**Примечание:** в файле также объявлены `createSoftwareSystemFocusLayout` и `createNonSoftwareSystemFocusLayout` (полукруг / кольцо соседей с полноценными силами), но **из `createForceSimulation` они не вызываются** — актуальная логика для продукта описана выше.

### 3.3. Синхронизация с React

При смене графа (`graphKey`) или размера canvas пересоздаётся симуляция. В **`GraphCanvas`** при `alpha(1).restart()` на каждом `tick` увеличивается `layoutVersion` и сбрасывается кэш геометрии рёбер; в режиме **heavy** симуляция принудительно останавливается по таймеру или при низком alpha.

## 4. От `SimNode` к `LayoutNode`

`simToLayout` преобразует массив `SimNode` в массив `{ node, x, y, width, height }` для текущего кадра. Это и есть **мировые координаты** диаграммы.

## 5. Геометрия рёбер

### 5.1. Дедупликация и тип линии

Для каждого ребра из `visibleEdges` строится полилиния; параллельные рёбра с одинаковой текстовой подписью (dedupe по `pair|label`) отбрасываются.

**Пунктир:** для типов Child / Deploy (`isStructuralEdgeRelationship`) — штрих-пунктир `EDGE_DOT_DASH` при отрисовке.

### 5.2. Режим «лучшее качество» (A*)

**`buildEdgeGeometry`:** между каждой парой узлов вызывается **`routeEdgeWithAStar`** (`astarGridRouter.ts`):

- Мир дискретизируется с шагом **`GRID_CELL` (60 px)**.
- Все карточки — препятствия с паддингом; путь ортогональный, поиск A* по сетке.
- Флаг **`horizontalFirst`** задаёт приоритет первого сегмента (горизонтально vs вертикально), если у одного из концов один лейбл узла.

Полилиния затем сглаживается при рисовании через **квадратичные кривые** (`traceSplinePath`).

### 5.3. Упрощённый режим (прямые отрезки)

**`buildEdgeGeometryStraight`:** две точки — выход с границы прямоугольника источника по направлению к центру цели и **вход** с границы цели. Без детуров.

Используется при **heavy** или **fast** режиме (см. ниже).

### 5.4. Подписи рёбер

- Текст: `graphEdgeLabel(edge)` (пусто для `Child`, для `Relationship` может подставляться technology).
- Позиция: **`labelPointAlongPolyline`** вдоль полилинии; смещение к концу **`labelBiasEnd(focusId, source, target)`** — чтобы подпись не перекрывала узел-фокус.
- **`resolveEdgeLabelOverlaps`:** жадное разведение по вертикали при пересечении bounding box’ов подписей (не в heavy).

### 5.5. Кэш геометрии

`GeometryCache` хранит: `layoutVersion`, `focusId`, массив полилиний, флаги пунктира, размещённые подписи.

Кэш **сбрасывается** при изменении `layoutVersion` (тик симуляции) или `focusId`. Пока совпадают — переиспользуются полилинии без пересчёта A*.

## 6. Режимы производительности (`GraphCanvas`)

Пороги по числу узлов `visibleNodes`:

| Режим | Узлов | Поведение |
|-------|-------|-----------|
| **best** | &lt; 50 | Полные A*-рёбра, подписи с разведением, сплайны с полным качеством. |
| **medium** | 50…100 | **fastMode:** прогрессивное построение рёбер по чанкам (`PROGRESSIVE_EDGE_CHUNK_SIZE`, бюджет времени); пока не готово — пустые рёбра; затем полная геометрия. Прямые рёбра вместо A* в этом режиме (`useStraightEdges`). |
| **strong** | &gt; 100 | **heavyMode:** прямые рёбра, без подписей рёбер, **без ореола** у линий (`hideHalo`), упрощённые сплайны при большом числе рёбер; симуляция может быть остановлена по таймеру (`HEAVY_MODE_MAX_TICKS`). |

Дополнительно: при числе рёбер ≥ `LOW_QUALITY_SPLINE_EDGE_THRESHOLD` включается упрощение полилинии для сплайна.

## 7. Отрисовка на Canvas (порядок слоёв)

Координаты экрана: `canvas` с учётом `devicePixelRatio`, затем **панорамирование** `panRef` и **масштаб** `zoomRef`.

**Порядок вызовов:**

1. **Заливка фона** палитрой `palette.canvasBg`.
2. **`paintDiagramNodes`** — все узлы:
   - **Кластер:** заливка, **пунктирная** рамка, заголовок `[GROUP …]`, подзаголовок; **без** внутренних мини-карточек на этом шаге.
   - **Обычный узел:** карточка, полоска типа, текст, теги, бейдж инцидентов.
3. **`paintDiagramEdges`** — для каждой полилинии: двойной обвод (halo + основная линия) или только линия в heavy; стрелка на конце; палитра задаёт цвета.
4. **`drawEdgeLabelPill`** для каждой размещённой подписи (не в heavy).
5. **`paintClusterInnerMiniCards`** — **только** внутренние прямоугольники и имена участников группы.

**Зачем два прохода для кластера:** рёбра рисуются **поверх** карточек. Если мини-карточки рисовать вместе с рамкой группы до рёбер, линии перекрывают ячейки. Поэтому ячейки выносятся **последним** слоем.

---

## 8. Тема оформления

`DiagramPalette` (`diagramTheme.ts`) задаёт фон, цвета рёбер, карточек, текста, кластера, бейджей и подписей рёбер для светлой и тёмной темы. Переключение темы вызывает перерисовку `draw` через `useEffect` по `palette`.

---

## 9. Экспорт

Общая геометрия для файла: **`computeFullExportData`** → `computeDiagramBounds` по layout, полилиниям и подсказкам подписей (с отступом `EXPORT_PAD`).

- **PNG:** offscreen canvas: тот же порядок, что и на экране (узлы → рёбра → подписи → внутренности кластеров).
- **SVG:** `buildDiagramSvgString` — фон, рёбра, текстовые подписи рёбер, узлы; для кластеров передаётся `clusters` с составом; **внутренние ячейки** рисуются отдельным блоком в конце, чтобы совпадать с логикой canvas.
- **JSON:** снимок узлов с координатами, рёбра, маршруты линий, подписи, метаданные.

---

## 10. Краткая схема потока данных

```
nodes/edges (props)
  → collapseDenseSimilarNodes (если есть выбор)
  → createForceSimulation → SimNode[] (мировые координаты)
  → simToLayout → LayoutNode[]
  → buildEdgeGeometry* (A* или прямые) + подписи
  → GeometryCache (при неизменном layoutVersion + focus)
  → paint: узлы → рёбра → подписи → мини-карточки кластеров
```
